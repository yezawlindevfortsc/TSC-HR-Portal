HR Portal · Technical Architecture

Architecture, Data Flow & the AI–PII Sandbox

How the system is built: a cost-aware sync off your existing HRIQ–to–Excel extraction, a Python-first stack, Singapore-resident hosting, and a hard wall between personal data and any AI model.

Batch sync, not paid API PDPA · SG residency AI sandboxed from PII Human-in-the-loop
01 / FOUNDATIONS

Five architectural decisions

Everything downstream follows from these. Two are shaped by your constraints — the HRIQ API cost and PDPA — and the rest enforce the human-accountability model from the requirements spec.

Sync, don't call

The Portal reads from its own database, kept current by scheduled batch extraction from HRIQ. The expensive live API is avoided except for rare, high-value writes.

Python-first

Backend in Python to reuse your existing extraction scripts and team familiarity. Your scripts become the ingestion backbone, not throwaway tooling.

AI is sandboxed

A model never sees raw personal data it doesn't need, and never performs statutory or financial math. A gateway scrubs every prompt that crosses the trust boundary.

Deterministic core

Payroll, levy, CPF, encashment and compensation run as tested Python code with traceable breakdowns — never an LLM.

SG-resident & sealed

Personal data is hosted in-region, the Portal is not publicly routable, and every action is written to an immutable audit log.

02 / DATA SOURCING

The HRIQ sync pattern

Your existing Python-to-Excel extraction is productionised into a scheduled pipeline. Extracted files land in a secure zone, are validated against a schema, then loaded into the Portal's operational store. The Portal queries that store — fast, and free of per-call API charges.

Cadence & freshness

  • Scheduled runs — e.g. nightly full sync plus intra-day deltas for time-sensitive data (permits, attendance, OT). Tunable per dataset.
  • Validation gate — each import is checked for schema, row counts and anomalies before it replaces live data; a failed import is quarantined, not loaded.
  • Staleness indicator — every dashboard shows "data as of <timestamp>" so HR never acts on stale figures unknowingly.

Write-back (when a change must return to HRIQ)

  • Default: the Portal generates a formatted import file (the same format HRIQ already accepts) for a human to upload — no API cost, human-gated.
  • Selective API: reserve paid API writes only for low-volume, high-value cases where file import is impractical. Metered and logged.
⚠ PDPA consequenceBecause the Portal keeps a synced copy of personal data, that database is squarely in PDPA scope. It must carry the same retention, access-control and breach obligations as HRIQ itself — see §6.

05 / THE SANDBOX

How AI is walled off from personal data

This is the part that protects you. The rule: the model only ever receives the minimum it needs, de-identified, through one auditable gateway — and it is never trusted with a calculation or a final decision.

The seven controls

  • 1 · Capability boundary. The model is used only for drafting, retrieval Q&A, summarisation, classification and triage. It is structurally unable to write to the DB or trigger an action.
  • 2 · Data minimisation. Each AI call passes only the fields that task needs — never a whole employee record.
  • 3 · De-identification / tokenisation. Names, FINs, passport and salary values are swapped for placeholders before crossing the boundary, and re-hydrated locally afterward. The model works on a skeleton.
  • 4 · Single gateway. Every AI call routes through one internal service that enforces 1–3, checks the caller's role and purpose, and logs the (redacted) prompt and response.
  • 5 · Region-locked, no-train model. Inference runs on an in-region endpoint under a data-processing agreement with no training/retention — never a consumer AI product.
  • 6 · RAG isolation. The knowledge base holds only regulations, circulars and SOPs (non-personal). Personal data is not embedded into the vector store.
  • 7 · Human review gate. Output is labelled AI-generated and reviewed by the responsible person before anything is issued, filed or actioned.
⚠ Residual risk to acceptEven sandboxed, AI drafts can be wrong or subtly off. The human review gate (control 7) is the real backstop — it is a process control, not just a technical one, and must be enforced in the UI (nothing leaves the Portal without a named approval).

6B / CV INTAKE

CV parsing — upload to structured record

CVs are the single input to recruitment; AI does the data entry. Whether a candidate uploads through the public application page or HR uploads an agency/walk-in CV, the file follows one pipeline: stored encrypted, parsed by AI into structured fields, written to the database, and scored for fit — no manual keying.

1
A CV is uploaded — by a candidate on the public application page (consent captured at source) or by HR for an agency / walk-in / referral CV (consent + source recorded).
2
File lands in the encrypted object store and is virus-scanned. The raw CV is personal data — it carries the retention clock from day one.
3
AI parser extracts name, contact, experience, role and skills (OCR for scans + entity extraction). Only the minimal document text needed crosses the AI gateway; the structured result returns to the trust zone.
4
Fields are written as a structured candidate record in PostgreSQL, linked to the stored file. Low-confidence fields are flagged.
5
Fit score is computed deterministically — extracted skills/experience matched against the role profile — and the candidate enters the screening pipeline, ranked.
6
Human reviews the extracted fields and the ranking before routing to a hirer. Extraction is assisted, not autonomous.
⚠ Two design constraintsAccuracy: CV layouts are messy; parsing is never 100%. The review-and-correct step stays, and low-confidence fields are flagged rather than trusted. Explainable ranking: the fit score is deterministic skill/experience matching — not an opaque LLM judgement — so a shortlisting decision can always be explained and defended (relevant to FCF / anti-discrimination record-keeping).

07 / WALKTHROUGHS

Two end-to-end flows

A · Generating a bond letter (AI-assisted)

1
Executive (Staff) selects an employee and "draft bond letter" in their workspace.
2
App fetches needed fields from the Portal DB (trust zone) — bond terms, dates.
3
AI Gateway de-identifies: real name/amount become [EMPLOYEE_1], [AMOUNT_1]; only the skeleton crosses to the model.
4
Model returns drafted letter prose with placeholders intact; gateway logs the exchange.
5
Template merge inside the trust zone re-inserts real values. Draft labelled "AI-generated".
6
Human review gate: Executive (and approver per DOA) reviews and releases. Issued version archived; action audited.

B · DRC / permit-expiry alert (AI-led monitoring)

1
Nightly extraction refreshes worker/permit data into the Portal DB.
2
Compliance monitor (deterministic) recomputes DRC headroom and scans permit/passport expiry windows.
3
Threshold breaches and upcoming expiries become prioritised alerts with owner + due date.
4
FW Executive sees them on the compliance board at login; the monthly DRC report is auto-compiled for the HR Manager.
5
Human acts: the Executive initiates renewal/correction; the Manager approves corrective action. No money or filing moves without a person.

Note flow B touches no LLM at all — the highest-value monitoring is pure deterministic code. AI is the assistant, not the engine.

08 / OPS & OPEN ITEMS

Environments & decisions still open

Environments & operations

  • Three environments — Dev, Staging (with synthetic/masked data), Production. No real PII in Dev.
  • Backups — automated daily, encrypted, with periodically tested restores.
  • Monitoring — uptime, job-failure alerts (a failed extraction must page someone), and audit-log review.
  • Parallel-run — payroll/calculation outputs reconciled against the current process before becoming authoritative (per the SRS phasing).
