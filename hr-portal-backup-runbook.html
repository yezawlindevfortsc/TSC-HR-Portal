Backup & Recovery Runbook

What is backed up, where it lives, and exactly what to do when something is lost — written so a non-developer can execute every recovery, with Claude in a session as co-pilot.

3-2-1 rule Tested restores, quarterly PDPA applies to backups too
00 / PRINCIPLES

Three rules this runbook lives by

  • 3-2-1: at least 3 copies of the data, on 2 different systems, 1 of them separate from the main cloud account — so even a full account compromise or ransomware event cannot touch every copy.
  • A backup you haven't restored is a hope, not a backup. We rehearse a real restore every quarter and log it (§04).
  • Backups are personal data. They're encrypted, access-restricted, and retention-limited so purged records also age out of the backups (§05).

These targets are deliberate: for a 6-user internal portal, a day of downtime is an inconvenience, not a crisis — so we buy simplicity rather than expensive zero-downtime engineering. The one thing we never accept losing is data.

Why the separate storage account mattersThe weekly exports go to a storage account with different access keys, immutability (write-once) enabled, and no connection from the portal app. If the main environment is compromised or encrypted by ransomware, the attacker cannot reach these copies. This is the "1" in 3-2-1 and the single most important line in this document.
02 / MONITORING

Knowing the backups actually ran

  • Automated: the weekly export job sends a success/failure email; a failed nightly Azure backup raises a Defender alert. Silence is not success — the job also sends a weekly "backup OK" heartbeat, so a missing email is itself the alarm.
  • Monthly (your routine, 2 min): glance at the backup dashboard — last PITR timestamp, last export date, export file size roughly consistent with the previous one (a sudden tiny file = something wrong).
03 / RECOVERY SCENARIOS

When something is lost — do this

Each scenario is written for you to execute with me in a session ("open Claude, open this page"). Times are realistic estimates including verification.

A · Someone deleted or overwrote a record / documentCommon · low risk~15–30 min · no downtime
  1. Don't panic and don't try to re-create it by hand — that complicates recovery.
  2. Document: Azure Portal → storage account → the container → toggle "Show deleted blobs" → select the file → Undelete. Done.
  3. Database record: open a session with me. We restore the database to a point just before the deletion as a copy, extract the affected record(s), and re-insert them into the live database. The live system never goes down.
  4. Note the incident in the drill log (§04) — real recoveries count as drills.
B · Database corrupted or badly wrong (bad import, bulk error)Rare · medium~1–2 h · portal offline during restore
  1. Stop the portal app (Azure Portal → App Service → Stop). Tell the HR team to hold off; manual process meanwhile.
  2. Identify the last-known-good time (the audit log or the HR team will know when things were right).
  3. Azure Portal → PostgreSQL server → Restore → pick date/time → restores to a new server (the damaged one is untouched — you can't make it worse).
  4. With me in a session: point the app at the restored server (one connection-string change in Key Vault), start the app, and run the verification checklist (record counts, latest documents, one letter generation).
  5. Anything entered between last-good and now is re-entered by HR from the day's paperwork — this is the ≤24 h RPO in practice.
C · Azure Singapore region outageVery rareHours (Azure's recovery) or ~2–4 h (geo-restore)
  1. First: check the Azure status page. Most outages resolve on their own — for a 1-day RTO, waiting is usually the right move.
  2. If it's prolonged: geo-redundant backups allow restoring the database and storage into another region (Southeast Asia pair). We do this together in a session. Note: data temporarily leaves Singapore — an accepted-in-emergency deviation; document it and restore to SG when the region returns.
D · Account compromise / ransomwareWorst case~1 day · assume main environment is hostile
  1. This is a security incident first — follow the Security Plan §06 (contain, preserve, notify) before any restore.
  2. Do not restore into the compromised environment. With IT admin + me: fresh resource group (or fresh subscription), rebuilt from git.
  3. Data comes from the separate storage account exports (untouchable by the attacker): latest pg_dump + document copies. Worst-case data loss: one week.
  4. All secrets re-issued from the Key Vault inventory; all users re-verified; pen-tester engaged before going back live.
E · Full rebuild from nothingThe ultimate test~1 day
  1. Everything needed exists in three places: git (all code + infrastructure config), backups (all data), Key Vault / secret inventory (all credentials).
  2. With me: deploy infrastructure from the config in git → restore database from the latest export → copy documents back → re-issue secrets → verify. This is rehearsed once a year as the annual drill (§04).
04 / DRILLS

Restore drills — the discipline

  • Quarterly (~30 min, with me): alternate between Scenario A (restore one record from PITR into a test copy) and Scenario B (full restore to a new server, verify, delete). We never drill against production.
  • Annually: one Scenario E rebuild into a throwaway environment — proves that git + backups + secrets inventory are genuinely sufficient to resurrect everything.

05 / PDPA & RETENTION

Backups hold personal data too

  • Same protection as production: encrypted at rest, access limited to the operator (you) and break-glass IT admin; access to the separate export account is itself logged.
  • Retention by design: PITR window 35 days, weekly exports kept 8 weeks. When a record is purged from the live system under the retention policy, it disappears from all backups within ~2 months automatically — no manual backup-scrubbing needed. This "purge + cycle-out" approach is the practical way to honour retention obligations in backups.
  • Restores can resurrect purged data: after any restore, re-run the retention job before going live so records purged since the backup date are purged again.
  • Geo-redundancy note: Azure's paired-region copies exist for disaster recovery; confirm the pairing and document it in the PDPA records as part of the data-location disclosure.
06 / SETUP CHECKLIST

One-time setup (Phase 0, with me)

  1. Enable automated backups + 35-day PITR + geo-redundancy on the PostgreSQL server.
  2. Enable blob versioning + 30-day soft delete + geo-redundant storage on the documents account.
  3. Create the separate backup storage account: different keys, immutability policy, no app access; store its credentials only in the operator's break-glass location.
  4. Deploy the weekly export job (database dump + document sync) with success/heartbeat emails to you.
  5. Key Vault: soft delete + purge protection on; write the secrets inventory.
  6. Git remote configured; confirm infra config is in the repo, not just on a machine.
  7. Book the first restore drill.
The honest failure modeBackup systems don't usually fail loudly — they fail by nobody noticing they stopped. The heartbeat email (§02) and the drill log (§04) exist precisely so that "it silently stopped working in March" cannot happen here.
