00 / HOW TO USE THIS

Who does what

Every item in this plan is tagged: CLAUDE means it's built into the code and configuration during development sessions — you don't maintain it, you just confirm it exists. YOU means it's part of your operating routine (§05) — short, scheduled, and written as checklists.

The security stance in one lineThe portal holds the personal data of ~300 staff and workers (identity documents, salaries, medical records). We defend it with identity-first access (SSO + MFA), a locked-down cloud setup with no public data surfaces, an AI layer that never sees raw personal data, and an audit trail on everything — then verify it with an independent pen test before real data goes in.
01 / THREAT MODEL

What we're protecting, and from whom

The assets: employee & worker PII (NRIC/FIN, passports, salaries, bank details, medical records, CVs), statutory records, and the company's compliance standing. A leak is a PDPC-reportable event and a reputational problem; a tampered payroll record is a financial one.

03 / AI-SPECIFIC SECURITY

Rules for the AI layer

Beyond the de-identification gateway already specified in the architecture, three rules cover the AI-specific attack surface:

  • Model output is data, never instructions. Whatever the model returns is displayed, validated, or stored — it can never trigger an action, call a tool, or write to the database directly. This makes prompt injection annoying rather than dangerous.
  • Uploaded documents are hostile until proven otherwise. CV parsing returns only schema-validated fields (name, contact, experience…); free text from inside a document never becomes part of a system instruction. The fit-score is deterministic code, so a CV cannot "talk its way" up the ranking — and a human reviews every extraction.
  • No personal data in prompts, no retention at the provider. Enforced by the gateway (placeholders in, re-hydration locally) and contractually by using a region-locked enterprise endpoint with no-training terms — never a consumer AI product.

05 / YOUR ROUTINE

The operator routine YOU

Monthly — 15 minutes

Sign-in review: Entra ID → Sign-in logs → filter HR Portal → scan for foreign locations, odd hours, failed-login bursts.
User list: confirm the HR security group contains exactly the current HR team — no leavers, no extras.
Defender for Cloud: open the recommendations page; anything red/new → bring to the next Claude session.
Storage check: confirm blob containers still say "Private" (30 seconds — this is the one that bites companies).
Backups: confirm last night's backup succeeded and the weekly export ran (see Backup Runbook).
Updates: apply the month's dependency-update change (I prepare it; you approve the deploy).

Quarterly — with me, in a session (~45 min)

Access review with the HR Manager: does each role's scope still match reality? Any coverage-mode grants that should have expired?
Restore drill (per the Backup Runbook) — and log it.
Audit-log sample: we pull a week of logs and eyeball for anomalies together.
Rotate any long-lived credentials that can't use managed identity.

Before real personal data enters — once

Independent security review + penetration test (external, ≈ S$8k–20k). The single professional engagement this plan retains — because the whole value of a pen test is eyes that didn't build the system.
IT admin confirms: MFA enforced, conditional access live, admin accounts hardened.
06 / IF SOMETHING GOES WRONG

Incident response — the first hour

  1. Contain. Suspected account compromise → IT admin disables that Entra account (one click). Suspected app/system compromise → stop the App Service (portal goes offline — acceptable; HR falls back to manual).
  2. Preserve. Don't delete anything. Note times, what you saw, screenshots. The audit log is append-only — it's your evidence.
  3. Assess with me. Open a session; we establish together what data, whose, and how much was exposed or altered.
  4. Notify per PDPA. If the breach is likely to cause significant harm or affects ≥500 individuals, notify the PDPC within 3 calendar days of assessing it as notifiable, and affected individuals where required. Inform your management immediately regardless.
  5. Recover. Restore per the Backup Runbook; rotate all credentials; document the root cause and the fix before switching back on.
Keep this reachablePrint this section. If the incident is a compromised laptop or account, you may not have normal access to find it.
07 / HONEST LIMITS

What this plan does not claim

  • I review my own work adversarially, but self-review is not independence — hence the mandatory external pen test before real data, and after any major scope expansion (e.g. payroll go-live).
  • Security is a practice, not a state. The routine in §05 is the plan — if the monthly 15 minutes stops happening, the plan is no longer being followed.
  • This document covers the portal. Company-wide security (laptops, email phishing, HRIQ itself) is outside its scope but affects it — an attacker who owns your laptop owns your portal session.
